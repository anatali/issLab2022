Finite State Machines

The behavior of a Finite State Machine (FSM) is based on the following concepts:
  1. A fsm-actor can be described as a 5-tuple (States, Inputs, Outputs, Transitions, InitialState):
    • States: Set of possible states (StateSpace)
    • Inputs: Set of possible input elements (called input alphabet; in our case messages)
    • Outputs: Set of possible output elements (called output alphabet; in our case actions )
    • InitialState: The initial state
    • Transition: A Transition defines the new state and output given the current state and input. It is often expressed by two functions: 
      nextState:      States x Inputs -> States
output:         States x Inputs -> Outputs
  2. A Moore's Finite State Machine generates output while it is in a particular state (output depends on state only). Thus, when the fsm-actor is in state, it first executes a sequence of terminating Actions and then checks for the possibility to 'fire' a transition among the set of the active transitions related to that state.
When a message msg is received, the actor checks for a possible transition for that msg in the current state.

If no transition is possible, the message msg is discarded if it is an event; otherwise it might be stored (the choice is up to the application designer) in a local msgQueue.

If the transition is possible, the actor goes into the next state related to that transition and then looks at empty moves or to transitions related to stored messages.

The actor attempts to manage all the stored messages before looking to a new message.

The Fsm class

abstract class  Fsm(  
	val name:  String,
	val scope: CoroutineScope = GlobalScope,
	val discardMessages : Boolean = false,
	val usemqtt  :    Boolean = false,
	val confined :    Boolean = false,
	val ioBound :     Boolean = false,
	val channelSize : Int = 50 ) : MqttCallback{

val mqtt  = MqttUtils(name)	
val fsmactor = scope.actor( dispatcher, capacity=channelSize ) {...}	
	...
}
  • discardMessages: when true, the unexpected messages are discarded;

  • usemqtt: when true, the Fsm will send/receive messages by using a MQTT broker;

  • confined: when true, the embedded fsmactor will use a single-Thread dispatcher;

  • ioBound: when true, the embedded fsmactor will use a 64-Thread dispatcher;

  • channelSize: when true, the embedded fsmactor will set that size for its channel;

Forward (built-in operation)

suspend fun forward(msgId : String,payload: String,dest : Fsm ){
  val msg = AppMsg.buildDispatch(
     actor=name, msgId=msgId , content=payload, dest=dest.name)
  forward( msg, dest )
}

suspend fun forward(msg : AppMsg,dest:Fsm ){
	if( ! dest.fsmactor.isClosedForSend) dest.fsmactor.send( msg  )
	else ...
}

Emit (built-in operation)

suspend fun emit(  msgId : String, payload: String ){
  val msg = AppMsg.buildEvent(
	actor=name, msgId=msgId , content=payload )
  emit( msg )
}

suspend fun emit(  msg : AppMsg ){
if( usemqtt ) {
	mqtt.publish( "unibo/qak/events", msg.toString() )
}else{ ... }
}

Messages

The Kotlin object utils/Messages.kt provides: 
suspend fun forward(  sender: String, msgId : String, 
	payload:String, dest:Fsm )  //forward1

suspend fun forward(  sender: String, msgId : String, 
	payload:String,destName:String, mqtt:MqttUtils ) //forward2

						
suspend fun emit(  sender: String, msgId : String, 
	payload:String, mqtt:MqttUtils ) //emit
  • forward1: builds an AppMsg with the given args and puts it into the queue of the given dest actor.

  • forward2: builds an AppMsg with the given args and publish it on the unib/qak/$destName topic using the given mqtt utility.

  • emit: builds an AppMsg with the given args and publish it on the unib/qak/events topic using the given mqtt utility.

The FSM language

The custom DSL language is implemented by the class Fsm.kt, as an example of a Kotlin internal DSL, as introduced in LabKotlin | Towards DSL

An example

Let us report here the quite simple example defined in demoDSL/demoDSL.kt: 
override fun getInitialState() : String{
 return "s0"
 }

override fun getBody() : (Fsm.() -> Unit){
 return {
	state("s0") {
		action {  
			println(" ... ")
		}
		transition( edgeName="t0",
			targetState="s1", cond=doswitch())	 //empty move		
	}			
	state("s1") {
		action {
			...
		}
		transition( edgeName="t1",
			targetState="s2",   cond=whenDispatch("msg1") )
		transition( edgeName="t2",
			targetState="s3", cond=whenDispatch("msg2") )
	}
	state("s2") {
		action {	...	}
 		transition( edgeName="t0",
			targetState="s3", cond=whenDispatch("msg2") )
	}
	state("s3") {
		action {	... 	}
		transition( edgeName="t0",
			targetState="s1", cond=doswitch())	 //empty move
	}
	}
Since the default of Fsm.kt is discardMessages=false, the unexpected messages are inserted in a local msgQueue. Thus, with the message sequence:
  • msg1, msg2: the final state is s1
  • msg2, msg1: the final state is s2
  • msg1, msg1, msg2: the final state is s2
If we launch the demo with discardMessages=true with
  • msg1, msg1, msg2: the final state is s1
